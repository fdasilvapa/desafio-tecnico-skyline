
### 1. Como você costuma estruturar seus commits no Git?

**Resposta:** Procuro usar os padrões de conventional commits para organização e rastreamento, por exemplo:
**feat -** Funcionalidades do sistema
```
# Exemplo
feat (contexto): Implementa funcionalidade de drag-and-drop nas colunas de tarefas.
```
  > Contexto - opcional, geralmente uso para especificar determinada seção - feat(navbar): - feat(footer): - feat(contact):

**fix-** Correções em bugs ou problemas
**docs -** Atualização ou adição de documentos relacionados ao desenvolvimento do sistema
**refactor -** Refatoração de código
> Esses são os que uso mais ativamente, mas sei que existem outros. Também é válido ressaltar que, principalmente em sistemas mais complexos, é uma ótima prática dar descrições detalhadas no pr (pull request) também, explicando as decisões e alterações que serão integradas.
  

### 2. O que você entende como Engenharia de Software? (Uma visão além do teórico)

**Resposta:** Engenharia de software engloba todos os processos do desenvolvimento de um sistema, desde o levantamento de requisitos até a entrega contínua.
Por isso, um bom desenvolvedor não pode se limitar a escrever um código funcional, ele precisa entender as dores do cliente para transformar isso numa solução viável.

> Um dos projetos que participo na faculdade está se provando complexo justamente por causa disso, eu e meu grupo temos uma ideia muito boa e não queremos um simples trabalho de conclusão, queremos transformar aquilo num produto de sucesso e, para isso, ainda precisamos alinhar corretamente a dor que iremos sanar com esse sistema.

Em resumo, engenharia de software é mais do que alguns passos a serem seguidos, é solucionar problemas complexos através da tecnologia, para gerar o máximo de valor possível no produto final.

### 3. Quais aplicações Web/App Mobile são referências pra você?

**Resposta:** De todos os que uso diariamente, eu diria que o primeiro que vem na cabeça é o Trello, desde que comecei a usá-lo achei muito mais intuitivo e de fácil aprendizado, caso comparado com semelhantes (como o Jira). Sua interface é simples e direta, com poucos cliques o usuário consegue criar o que tinha em mente. 
Isso é um demonstrativo claro de como UI/UX influenciam na percepção de quem usa seu sistema,  telas amigáveis, com ícones de fácil reconhecimento, fluxo de ações dentro da aplicação, tudo isso conta para uma experiência melhor do usuário final.

### 4. Você usa de ambientes de desenvolvimento integrados com IA? (Ex: GitHub Copilot/Google Antigravity)

**Resposta:** Por enquanto como estou em fase de aprendizado, estou tentando evitar IAs integradas, como por exemplo as de auto completar código, para ter um pouco mais de contato escrevendo códigos na mão, porém uso bastante LLMs como o Gemini para auxiliar no desenvolvimento desses códigos. 
> Para meus próximos projetos de estudo, planejo não usar IA para gerar código, apenas para outros tipos de tarefas.

### 5. Como você descreveria o uso responsável de IA em empresa?

**Resposta:** As IAs são ferramentas extremamente importantes para aumentar a produtividade, porém se o desenvolvedor não souber o mínimo possível da teoria por trás de um software (lógica, arquitetura, UI/UX, etc.), as alucinações das IAs podem acabar sendo prejudiciais ao projeto que ele está desenvolvendo.
Sendo assim, é bom entender **o que**, **como** e o **porque** a IA está fazendo, além de conhecimentos sobre o próprio funcionamento das LLMs, como gerenciamento de contexto.
> Quanto maior a janela de contexto de um agente de IA, maior a chance de ela começar a alucinar nas respostas.
